# Библиотечная система на PostgreSQL

Учебный проект для отработки JOIN, транзакций, оконных функций, индексов и оптимизации запросов.

## Стек

- PostgreSQL 18
- psql / pgAdmin
- Git

## Содержание проекта

### Схема данных

- Таблицы `authors` и `books`
- Связь один-ко-многим через внешний ключ `author_id`
- Ограничения: `NOT NULL`, `DEFAULT`, `CHECK` (год публикации > 1449)
- Типы данных: `SERIAL`, `TEXT`, `INTEGER`, `BOOLEAN`, `TIMESTAMPTZ`

### Запросы

- `INNER JOIN` — вывод книг с именами авторов
- `LEFT JOIN` — вывод всех авторов, включая тех, у кого нет книг
- `GROUP BY` + `COUNT` — количество книг по каждому автору
- `UPDATE` с `RETURNING` — изменение статуса выдачи книги

### Транзакции

- `BEGIN`, `COMMIT`, `ROLLBACK`
- Демонстрация отката изменений при ошибке в `WHERE`
- Понимание поведения транзакции при разрыве соединения
- Практические эксперименты с двумя консолями:
  - Грязное чтение запрещено
  - Неповторяющееся чтение на `READ COMMITTED`
  - Фантомное чтение
  - `REPEATABLE READ` против `READ COMMITTED`
  - Аномалия сериализации на `SERIALIZABLE`
- Блокировки строк при незавершённых транзакциях
- Ошибки: `SET TRANSACTION ... после BEGIN`, прерванные транзакции

### Индексы и производительность

- Создание индекса: `CREATE INDEX idx_title ON books (title)`
- Сравнение `Seq Scan` и `Index Scan` через `EXPLAIN ANALYZE`
- Разница между `Index Scan`, `Bitmap Heap Scan` и `Index Only Scan`
- Покрывающий индекс с `INCLUDE`

### Оконные функции

- `ROW_NUMBER()` — нумерация книг внутри каждого автора
- `RANK()` — нумерация с пропуском номеров при дубликатах
- `AVG(...) OVER(PARTITION BY ...)` — средний год публикации по автору для каждой книги
- `LAG()` / `LEAD()` — предыдущее и следующее издание книги
- `FIRST_VALUE()` / `LAST_VALUE()` — первое и последнее издание книги
- Вычисление разницы между годами изданий через `LAG`

### CTE (Common Table Expressions)
- Временные таблицы в рамках одного запроса (`WITH ... AS`)
- Несколько CTE через запятую
- Сравнение CTE с подзапросами: когда что лучше

### Подзапросы
- `IN` — проверка вхождения в список
- `EXISTS` — проверка наличия хотя бы одной строки (останавливается после первого совпадения)
- `ANY` / `ALL` — сравнение с любым / со всеми значениями из подзапроса
- Разница между `EXISTS` и `IN` с точки зрения производительности

### Нормальные формы (проектирование БД)
- 1NF (Первая нормальная форма): атомарные значения, никаких списков и массивов в одной ячейке
- 2NF (Вторая нормальная форма): нет частичных зависимостей от составного ключа
- 3NF (Третья нормальная форма): нет транзитивных зависимостей (неключевые поля зависят только от первичного ключа)

### Что сделано в проекте:
- Связь многие-ко-многим (книги ←→ авторы) через таблицу `book_authors`
- Отдельная таблица `editions` для поддержки нескольких изданий одной книги
- Текущая таблица `books` нормализована для одного издания, но готова к расширению
- Таблица authors то-же не отвечает стандартам 2nf и 3 nf, вернусь к ней после изучения других тем
  
### Миграция данных (переход на нормализованную схему)
- Создана новая таблица `books` с уникальными названиями
- Созданы таблицы `book_authors` (связь многие-ко-многим) и `editions` (издания)
- Через временные колонки `new_book_id` выполнено обновление внешних ключей
-  Использован `UPDATE` с `JOIN` для связывания старых и новых `book_id` по названию книги

### Что такое MVCC(Multi-Version Concurrency Control) — механизм, который позволяет читать и писать одновременно без блокировок. Каждое изменение строки не перезаписывает её, а создаёт новую версию. Старые версии остаются (как «тень»). Системные колонки (невидимые, но существующие)
- `xmin` — ID транзакции, которая создала эту версию строки
- `xmax` — ID транзакции, которая удалила (или изменила) эту версию строки
- `ctid` — физический адрес строки на диске (блок, позиция)

### WAL (Write-Ahead Log) Журнал предзаписи, в который изменения записываются до применения к основной базе данных.
Зачем нужен:
- Восстановление после сбоя (Crash Recovery)
- Восстановление на момент времени (PITR)

### Роли и привилегии
- Создание ролей с логином (`CREATE ROLE ... WITH LOGIN`)
- Выдача прав (`GRANT SELECT, INSERT ...`)
- Отзыв прав (`REVOKE ...`)
- Групповые роли для управления правами
- Просмотр прав через `\dp` и `\du+`
- Разница между `LOGIN` и `SUPERUSER`
- Удаление ролей

### Бэкапы и восстановление
- Логический бэкап одной базы (`pg_dump`)
- Бэкап всех баз и ролей (`pg_dumpall`)
- Сжатый дамп в собственном формате (`-Fc`)
- Бэкап только схемы (`-s`) или только данных (`-a`)
- Восстановление из SQL-дампа (`psql -f`)
- Восстановление из сжатого дампа (`pg_restore`)
- Восстановление одной таблицы (`-t`)
- Почему обычное копирование файлов (`cp`) не подходит для бэкапа работающей БД

### JSONB в PostgreSQL
- Хранение и извлечение данных из JSONB (`->`, `->>`)
- Фильтрация по полям JSONB (`@>`, `?`)
- Поиск значений в массивах JSONB
- Создание GIN индекса для ускорения JSONB-запросов
- Обновление JSONB: добавление и удаление полей (`||`, `-`)

## Как развернуть проект

### Вариант 1 (чистый старт — финальная схема)
Выполнить файлы из папки `schema/` в порядке нумерации, затем `data/` для заполнения.

### Вариант 2 (увидеть всю историю миграции)
Выполнить файлы из `migrations/` по порядку:
1. `001_initial_schema.sql` — создаёт исходную плоскую схему
2. `002_normalize_books.sql` — преобразует её в нормализованную

После миграции структура будет идентична той, что в `schema/`.
